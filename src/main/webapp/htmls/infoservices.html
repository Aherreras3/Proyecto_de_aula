Definición

Los servicios web se se les puede considerar como aplcaciones web que estan esperandoa ser invocadas a travez de la red.
A travez de estos servicios web otras apliaciones pueden obtener informacion y trabajar en conjunto para mejorar la flexibilidad de la aplicacion sin importar su arquitectura o lenguaje en el que se haya realizado.
Entre las arquitecturas mas populares de servios web twnemos la SOAP y la REST, a continuación describiremos estas y algunas de sus caracteristicas mas importantes.

SOAP

El Protocolo simple de acceso a objetos (SOAP) es un protocolo de mensajería que permite que las aplicaciones se comuniquen mediante HTTP y XML. Representa un paradigma de intercambio de mensajes unidireccional, fundamentalmente sin estado, entre nodos. Mediante la combinación de intercambios unidireccionales con funciones proporcionadas por el protocolo de transporte y/o la aplicación subyacente información específica, SOAP se puede utilizar para crear interacciones más complejas como solicitud/respuesta, petición/respuesta múltiple, etc. (Mumbaikar & Padiya, 2013; Halili, Rufati & Ninka, 2013) El proceso de invocación de servicios web es muy importante; por lo que se establece el protocolo SOAP para intercambiar mensajes entre los proveedores de servicios y los consumidores. Es un formato de mensaje XML estructurado para intercambiar datos en un entorno distribuido. Utiliza un protocolo de transporte subyacente (HTTP, SMTP, etc.) a través del enlace. Por el Al momento de escribir este documento, hay dos versiones de SOAP: SOAP versión 1.1 y SOAP versión 1.2 que tiene trajo algunos beneficios nuevos: es más limpio, más rápido, tiene una mejor integración web y es más versátil.
Existen tres tipos de nodos SOAP:

  • Remitente SOAP: genera y transmite un mensaje SOAP.
  • Receptor SOAP: recibe y procesa el mensaje SOAP y también puede generar una respuesta SOAP, mensaje o falla como resultado.
  • Intermediario SOAP (Reenvío o activo): es tanto un receptor SOAP como un remitente SOAP. recibe y procesa los bloques de encabezado SOAP dirigidos a él y reenvía el mensaje SOAP hacia un SOAP receptor.

SOAP es un protocolo independiente ligero. es independiente y liviano porque no importa de qué sistema operativo o plataforma se use el servicio: responde de la misma forma en cualquier plataforma o sistema operativo. Todo esto es posible gracias a los protocolos XML y HTTP. Hay dos tipos de solicitudes de mensajería SOAP: Llamada a procedimiento remoto (RPC) y solicitud de documento.

Arquitectura REST

REST - (Transferencia de estado representacional) como su nombre lo indica, tiene que ver con la relación entre el cliente y el servidor y cómo se almacena el estado. La arquitectura REST se basa en el estilo de arquitectura cliente/servidor. Así, las solicitudes y las respuestas se construyen a partir del proceso de transferencia de los recursos. Todos los recursos están identificados por un único Identificador uniforme de recursos (URI), que normalmente representa un documento que captura el estado del recurso. En general, la arquitectura de estilo REST es mucho más ligera en comparación con SOAP. No requiere formatos como encabezados que se incluirán en el mensaje, como se requiere en la arquitectura SOAP. Por otro lado, analiza JSON: un lenguaje legible por humanos diseñado para permitir el intercambio de datos y facilitar su análisis y uso por parte del computadora. Se estima que es unas cien veces más rápido que XML.

Ventajas y Desventajas de REST

Esta arquitectura tiene si ventas y desventajas en comparacion con cotras arquitecturas entre las mas importantes tenemos:
VENTAJAS:

  • REST utiliza un formato de mensaje más pequeño y proporciona rentabilidad a lo largo del tiempo y un mejor rendimiento por los mensajes JSON con los que se realiza la comunicación. y no hay intensivo procesamiento requerido
  • Se reduce la curva de aprendizaje.
  • Soporta comunicación sin estado.
  • Es fácil de aprender e implementar.
  • Usa eficientemente verbos HTTP.
  • Ancho de banda ligero ya que su mensaje de paso es JSON.
  • (Notación de objetos de JavaScript) formato también puede utilizar muchos otros formatos.
  • REST puede ser consumido por cualquier cliente.
  • Hace que los datos estén disponibles como recurso..

DESVENTAJAS:

  • No es adecuado para una gran cantidad de datos.
  • Comparado con SOAP no cubre todas las variedades de estándares de servicios web como Seguridad, Transacciones, etc..
  • REST no es confiable.
  • Las solicitudes REST (especialmente GET) no son adecuadas para una gran cantidad de datos.
  • Latencia en los tiempos de procesamiento de solicitudes y uso de ancho de banda.
  • Las API REST terminan dependiendo de los encabezados para el estado (como para enrutar las solicitudes posteriores al mismo servidor de back-end que manejó la actualización anterior o para la autenticación).

SOAP vs REST

SOAP

Cambio de servicios en el aprovisionamiento web SOAP a menudo significa un cambio de código complicado en el lado del cliente.

SOAP tiene una gran carga útil en comparación con REST.

Requiere análisis de archivos adjuntos binarios..

SOAP no es una infraestructura inalámbrica amigable..

Los servicios web SOAP siempre devuelven datos XML..

Consume más ancho de banda porque una respuesta SOAP podría requerir más de 10 veces más bytes en comparación con REST.

La solicitud SOAP utiliza POST y requiere la creación de una solicitud XML compleja que dificulta el almacenamiento en caché de respuestas.

Los usos de SOAP Las API basadas en HTTP se refieren a las API que se exponen como uno o más URI de HTTP y las respuestas típicas están en XML/JSON. Los esquemas de respuesta son personalizados por objeto.

Independiente del idioma, la plataforma y el transporte.

Diseñado para manejar entornos informáticos distribuidos..

Más difícil de desarrollar, requiere herramientas.

Es el estándar prevaleciente para los servicios web y, por lo tanto, cuenta con un mejor soporte de otros estándares (WSDL, WS) y herramientas de los proveedores.






REST

El cambio de servicios en el aprovisionamiento web REST no requiere ningún cambio en el código del lado del cliente.

REST es definitivamente liviano, ya que está destinado a la transferencia de datos livianos a través de una interfaz más conocida: el URI.

Es compatible con todos los tipos de datos directamente.

REST es una infraestructura inalámbrica amigable.

Mientras que los servicios web REST brindan flexibilidad con respecto al tipo de datos devueltos.

Consume menos ancho de banda porque su respuesta es ligera.

Las API tranquilas se pueden consumir mediante solicitudes GET simples, los servidores proxy intermedios/proxies inversos pueden almacenar en caché su respuesta muy fácilmente.

REST, por otro lado, agrega un elemento de uso de URI estandarizados, y también da importancia al verbo HTTP utilizado (es decir, GET / POST / PUT, etc.

Independiente del lenguaje y la plataforma.

Asume un modelo de comunicación punto a punto, no para un entorno informático distribuido donde el mensaje puede pasar por o más intermediarios.

Mucho más simple de desarrollar servicios web que SOAP.

Falta de soporte de estándares para seguridad, políticas, mensajería confiable, etc., por lo que los servicios que tienen requisitos más sofisticados son más difíciles de desarrollar.
